## Why bother

Roon users have been asking for a more capable API for years. The official
[`node-roon-api`](https://github.com/RoonLabs/node-roon-api) is deliberately limited — no
metadata editing, no real library management, just a sandboxed slice of what the app can do.

So this is the brute-force route: watch what the desktop client *itself* sends to the Core,
figure out the wire format, and see how much of it you can replay from your own code. It's
an experiment, not a product, and definitely not blessed by Roon.

:::note[The one interesting finding]
On the local network, the protocol doesn't seem to use any authentication. The thing I
*expected* to be the wall — "mutations need a cloud auth token" — turned out (in my testing)
to be wrong. A correctly-formed call against a live object reference just works. The actual
hard part was the wire format and the fact that object references are ephemeral.
[How I got there →](/journey/)
:::

:::caution[Stability & safety — read before you run it]
This is an **internal protocol meant for Roon's own trusted code**, not a public API.
**It will break:** Roon versions client and Core in lockstep and changes it freely, so any
update can break this client. **Misuse can degrade your own Core:** the protocol isn't
hardened for third parties, and there are resource-management obligations across the
connection (releasing handles, tearing down subscriptions, completing callbacks) that a
reverse-engineered client may not get right — getting them wrong can leak memory or compute
*inside your own Roon Core*. Only point it at a Core you own and can restart. Roon's CTO
[made these same points on the forum](https://community.roonlabs.com/t/reverse-engineering-the-roon-desktop-clients-local-protocol-typescript-client-docs/321731).
:::

## What seems to work

This is one person tinkering against a single Core, confirmed by hand — not a test matrix
across setups. Treat everything as "worked for me," not "guaranteed."

<CardGrid>
  <Card title="Protocol" icon="document">
    Framing, varints, primitives, method calls, by-val structs, and the object graph are
    mapped well enough to talk to a Core. Cross-checked against my own packet captures.
  </Card>
  <Card title="A TypeScript port" icon="seti:typescript">
    ~14k LOC. Typed wrappers are code-generated for all ~1550 methods, but the vast majority
    are untested — generated from metadata, not exercised live.
  </Card>
  <Card title="Tried by hand" icon="approve-check">
    Reading the object graph, favoriting, playing an album, pause, standby, and a reversible
    metadata edit each worked once or twice against my setup.
  </Card>
  <Card title="Lots still open" icon="puzzle">
    Most notably full streaming-catalog search (arbitrary Tidal/Qobuz terms). In-library
    search half-works. Plenty of rough edges. [Pitch in →](/contributing/)
  </Card>
</CardGrid>
